# REQ-010: KafGraph Compliance Hub — GDPR Framework

## Summary

KafGraph shall provide a compliance engine that evaluates the knowledge graph
against regulatory frameworks (GDPR, SOC 2, EU AI Act), stores results as graph
nodes, and exposes them via API and dashboard.

## Motivation

Agent collaboration teams processing personal data need continuous compliance
monitoring. The compliance hub transforms KafGraph from a knowledge graph into
an auditable compliance platform, enabling DPOs to track GDPR obligations
alongside agent collaboration data.

## Requirements

### Compliance Engine (Phase 1)

- **REQ-010.1**: A `compliance.Engine` shall register `Rule` implementations and
  execute scans that evaluate rules against the graph.
- **REQ-010.2**: Rules shall be defined as Go structs (hardcoded) or YAML files
  (extensible).
- **REQ-010.3**: Scan results shall be stored as `ComplianceScan`,
  `ComplianceEvaluation`, `ComplianceFramework`, and `ComplianceRule` graph nodes
  with appropriate edges (`PART_OF_SCAN`, `EVALUATED_BY`, `DEFINES_RULE`, `SCOPED_TO`).
- **REQ-010.4**: Compliance scoring shall weight rules by severity:
  critical=3, high=2, medium=1, low=0.5.
- **REQ-010.5**: The engine shall support on-demand scans via API and optional
  scheduled scans via configuration.

### GDPR Modules (Phase 2)

- **REQ-010.6**: Thirteen hardcoded GDPR rules shall cover: DPO designation
  (Art. 37), legal basis (Art. 6), retention periods (Art. 30), data categories
  (Art. 30), TOMs (Art. 32), DSR timeliness (Art. 12), DSR documentation
  (Art. 15-22), 72-hour breach notification (Art. 33), special category breach
  notification (Art. 34), DPIA for high-risk processing (Art. 35), DPIA risk
  identification (Art. 35), processor contracts (Art. 28), and evidence for
  accountability (Art. 5(2)).
- **REQ-010.7**: CRUD endpoints under `/api/v2/compliance/gdpr/` shall manage:
  OrgSetup, DataCategory, LegalBasis, SecurityMeasure, ProcessingActivity,
  DataSubjectRequest, DataBreach, DPIA, DPIARisk, DataProcessor, ChecklistItem,
  and Evidence nodes.
- **REQ-010.8**: A DSR SLA endpoint shall calculate days remaining and overdue
  status for each active data subject request (30-day deadline per Art. 12(3)).
- **REQ-010.9**: Demo seed data shall exercise all GDPR modules with realistic
  scenarios including intentional compliance gaps for rule evaluation.

### Dashboard (Phase 2)

- **REQ-010.10**: A compliance dashboard view in the management UI shall display:
  score rings per framework, module breakdowns, latest scan results, and scan
  trigger button.
- **REQ-010.11**: Sub-views shall provide module-specific interfaces:
  RoPA registry, DSR tracker with SLA indicators, breach register with timeline,
  DPIA with risk matrix, and organisation setup form.

### Configuration

- **REQ-010.12**: `compliance.enabled` (default true), `compliance.rules_dir`
  (default "./rules"), `compliance.scan_interval` (default "24h"), and
  `compliance.auto_scan` (default false) configuration keys shall control
  the compliance engine.

## YAML Rule Extension Format

```yaml
id: GDPR-CUSTOM-001
framework: gdpr
module: ropa
article: "Art. 30"
title: "Custom retention check"
severity: high
check:
  nodeLabel: ProcessingActivity
  requiredProperties:
    - retentionPeriod
  condition: "property_not_empty"
```
